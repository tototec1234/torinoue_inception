| Q | 形式 | 判定 | 要点 |
|---|------|------|------|
| Q1 | 選択 | ✅ 正解 | `host not found in upstream` → コンテナ名が DNS に存在しない |
| Q2 | 記述 | ✅ ほぼ正解 | 核心は「conf を変えずに名前解決だけ満たす」割り切り |
| Q3 | 選択 | ✅ 正解 | 502 の主因は FastCGI 上流未到達。`--insecure` 補足も的確 |
| Q4 | 記述 | ⚠️ 部分正解 | メカニズムは正確だが合格理由は「TLS スコープ限定」。「内部転送確認でOK」はずれ |
| Q5 | 選択 | ✅ 正解 | ユーザ定義ネットワーク → コンテナ名 DNS |
| Q6 | 選択 | ✅ 正解 | `--insecure` の実際の curl 出力添付が優秀 |
| Q7 | 記述 | ⚠️ 部分正解 | 「複数ネットワーク所属」は正しい。「リフレッシュ」は根拠なし。核心は「起動済みコンテナには `docker run --network` が使えない」 |
| Q8 | 記述 | ✅ ほぼ正解 | 静的 vs FastCGI の切り分けは正確。「先が間違っている」→「先が応答しない」が正確 |

| Q11 | 記述 | ❌ 不正解 | 「未実装だからわからない」ではなく「設計上 WordPress は TCP 経由で `DB_HOST=mariadb` に繋ぐ → `wpuser@'%'` が効く」と断言できる必要あり |
| Q12 | 選択 | ✅ 正解 | root でのソケット接続が DB 生存確認に十分 |
| Q13 | 記述 | ⚠️ 部分正解 | 名前解決の点は合っている。不足点: NGINX が DB に直接触らなくても全員同一ネットワークに載せて compose トポロジを再現する意義 |

**弱点（レビュー前に重点復習）:**
- **Q11**: 「テスト経路（ソケット）と本番経路（TCP）の違い」を**設計レベルで言語化**する。WordPress は `DB_HOST=mariadb` で TCP 接続する設計なので `wpuser@'%'` で問題ない、と断言できること。参照: `session_logs/0011_session_log_inception.md` の Spike 記録「Inception での影響」節。
- **Q7**: `docker network connect` の正確な用途は「起動済みコンテナへのネットワーク追加」。「リフレッシュ」という概念は存在しない。
