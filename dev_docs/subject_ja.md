## Chapter II — はじめに

このプロジェクトは、Dockerを使ってシステム管理の知識を広げることを目的としています。あなた自身の新しい仮想マシン上で、複数のDockerイメージを仮想化して作成します。

## Chapter III — 一般的なガイドライン

- このプロジェクトは**仮想マシン上**で行う必要があります。
- プロジェクトの設定に必要なすべてのファイルは `srcs` フォルダに配置すること。
- **Makefile** も必要で、ディレクトリのルートに配置すること。Makefileはアプリケーション全体をセットアップする（= `docker-compose.yml` を使用してDockerイメージをビルドする）ものでなければなりません。
- この課題では、あなたのバックグラウンドによっては、まだ学んでいない概念を実践する必要があります。そのため、Docker の使い方に関するドキュメントや、この課題を完了するために役立つあらゆる資料を、ためらわずに多く読むことをお勧めします。

## Chapter V — 必須パート

このプロジェクトは、特定のルールのもとで異なるサービスで構成される小さなインフラストラクチャを構築するものです。プロジェクト全体は仮想マシン内で行い、**docker compose** を使用すること。

- 各Dockerイメージは、対応するサービスと**同じ名前**でなければならない。
- 各サービスは**専用のコンテナ**で動作すること。
- パフォーマンス上の理由から、コンテナは **Alpine** または **Debian** の**最新から1つ前の安定版（penultimate stable version）** をベースにビルドすること。どちらを使うかは自由。
- 自分で **Dockerfile** を書くこと（サービスごとに1つ）。Dockerfileは `docker-compose.yml` 内で Makefile から呼び出されること。
- つまり、プロジェクトのDockerイメージは自分自身でビルドする必要がある。**既成のDockerイメージの pull は禁止**。DockerHub等のサービスの利用も禁止（Alpine/Debianはこのルールの例外）。

### 構築するもの

- **NGINX** を含むDockerコンテナ（**TLSv1.2 または TLSv1.3 のみ**対応）
- **WordPress + php-fpm** を含むDockerコンテナ（インストール・設定必須）、**nginx なし**
- **MariaDB** のみを含むDockerコンテナ、**nginx なし**
- WordPressデータベースを格納する**ボリューム**
- WordPressのウェブサイトファイルを格納する**2つ目のボリューム**
- この2つの永続ストレージには **Docker named volumes** を使用すること。**バインドマウントは禁止**。
- 両方の named volume は、ホストマシンの `/home/<login>/data` にデータを保存すること。`<login>` は自分のユーザー名に置き換える。
- コンテナ間の接続を確立する **docker-network**
- コンテナは**クラッシュ時に自動で再起動**すること。

### 重要な注意事項

> **Dockerコンテナは仮想マシンではありません。** そのため、`tail -f` などのハック的な方法でコンテナを動かし続けるのは推奨されません。デーモンの仕組みと、それを使うべきかどうかについて読んでください。

- `network: host`、`--link`、`links:` の使用は**禁止**。
- `docker-compose.yml` ファイルに network の行が存在しなければならない。
- **無限ループを実行するコマンドでコンテナを起動してはならない。** これは entrypoint として使用されるコマンドや、entrypoint スクリプト内のコマンドにも適用される。以下は禁止されるハック的な方法の例：`tail -f`、`bash`、`sleep infinity`、`while true`

> PID 1 と Dockerfile のベストプラクティスについて読むこと。

### WordPress データベースの要件

- WordPressデータベースには**2人のユーザー**が必要。そのうち1人は管理者。
- 管理者のユーザー名に **admin/Admin** や **administrator/Administrator** を含めてはならない（例：admin、administrator、Administrator、admin-123 などは不可）。

> ボリュームはホストマシンの `/home/<login>/data` フォルダで利用可能になります。`<login>` は自分のものに置き換えてください。

### ドメイン名の設定

- ドメイン名がローカルIPアドレスを指すように設定すること。
- ドメイン名は **`<login>.42.fr`** とする。自分のログイン名を使用すること。
  - 例：ログインが `wil` なら、`wil.42.fr` が wil のウェブサイトを指すIPアドレスにリダイレクトされる。

### その他の禁止事項・義務

- **`latest` タグは禁止。**
- **Dockerfile にパスワードを含めてはならない。**
- **環境変数の使用が必須。**

> また、環境変数を保存するために **`.env` ファイル**を使用することが必須です。機密情報の保存には **Docker secrets** の使用が強く推奨されます。Gitリポジトリ内で（適切に設定されたsecrets 以外に）認証情報、APIキー、パスワードが見つかった場合、**プロジェクトは不合格**になります。

> **NGINX コンテナが、ポート443のみを使用し、TLSv1.2 または TLSv1.3 プロトコルでインフラストラクチャへの唯一のエントリーポイントでなければならない。**

### 期待されるディレクトリ構造の例

```
.
├── Makefile
├── secrets/
│   ├── credentials.txt
│   ├── db_password.txt
│   └── db_root_password.txt
└── srcs/
    ├── docker-compose.yml
    ├── .env
    └── requirements/
        ├── bonus/
        ├── mariadb/
        │   ├── conf/
        │   ├── Dockerfile
        │   ├── .dockerignore
        │   └── tools/
        ├── nginx/
        │   ├── conf/
        │   ├── Dockerfile
        │   ├── .dockerignore
        │   └── tools/
        ├── tools/
        └── wordpress/
```

`.env` の例：

```
DOMAIN_NAME=wil.42.fr
# MYSQL SETUP
MYSQL_USER=XXXXXXXXXXXX
[...]
```

> セキュリティ上の理由から、認証情報、APIキー、パスワードなどは**ローカルに保存**し、**gitで無視**すること。公開されたリポジトリに認証情報がある場合、プロジェクトは直接不合格になります。

> ドメイン名などの変数は `.env` ファイルに保存できます。

## Chapter VI — README の要件

Gitリポジトリのルートに **`README.md`** ファイルを置くこと。プロジェクトに不慣れな人（ピア、スタッフ、採用担当者など）がプロジェクトの概要、実行方法、トピックに関する詳細情報の場所をすぐに理解できるようにするためのものです。

`README.md` には少なくとも以下を含めること：

1. **最初の行**はイタリック体で次のように記載：  
   *This project has been created as part of the 42 curriculum by \<login1\>[, \<login2\>[, \<login3\>[...]]]*
2. **「Description」セクション** — プロジェクトの目標と概要を含めて明確に紹介
3. **「Instructions」セクション** — コンパイル、インストール、実行に関する情報
4. **「Resources」セクション** — トピックに関連する標準的な参考資料（ドキュメント、記事、チュートリアルなど）と、**AIの使用方法の説明**（どのタスク・どの部分に使ったか）

> 追加セクションがプロジェクトによって必要になる場合がある（使用例、機能一覧、技術的選択など）。必要な追加は明示的に記載される。

5. **「Project description」セクション** — Dockerの使用とプロジェクトに含まれるソースについて説明すること。主な設計上の選択と、以下の比較を含めること：
   - **仮想マシン vs Docker**
   - **Secrets vs 環境変数**
   - **Docker Network vs Host Network**
   - **Docker Volumes vs バインドマウント**

> **README は英語で書くこと。**

## Chapter VII — バリデーションの前提条件

既存の要件に加えて、リポジトリのルートに以下のドキュメントファイルを配置すること。Markdown形式（`.md`）で書くこと。

### USER_DOC.md — ユーザードキュメント

エンドユーザーまたは管理者が以下を理解できるよう、明確で簡潔に記述すること：

- スタックが提供するサービスの内容
- プロジェクトの起動と停止の方法
- ウェブサイトと管理パネルへのアクセス方法
- 認証情報の場所と管理方法
- サービスが正常に動作していることの確認方法

### DEV_DOC.md — 開発者ドキュメント

開発者が以下を実行できるように記述すること：

- ゼロからの環境構築（前提条件、設定ファイル、シークレット）
- Makefile と Docker Compose を使用したプロジェクトのビルドと起動
- コンテナとボリュームの管理に使用する関連コマンド
- プロジェクトデータの保存場所と永続化の仕組み

## Chapter IX — 提出とピア評価

通常通り、Gitリポジトリに課題を提出してください。ディフェンス（評価）で評価されるのは**リポジトリ内の成果物のみ**です。フォルダ名やファイル名が正しいか、再確認することをためらわないでください。

評価中に、プロジェクトの**簡単な修正が求められる場合があります**。これは、小さな動作変更、数行のコードの記述・書き直し、または簡単に追加できる機能などです。

このステップはすべてのプロジェクトに適用されるわけではありませんが、評価ガイドラインに記載されている場合は**準備しておく**こと。

このステップは、プロジェクトの特定の部分に対する**あなたの実際の理解度を確認する**ためのものです。修正は任意の開発環境で行うことができ、特定の時間枠が定義されていない限り、**数分以内に実行可能**であるべきです。

例えば、関数やスクリプトの小さな更新、表示の変更、新しい情報を保存するためのデータ構造の調整などを求められることがあります。

詳細（範囲、対象など）は評価ガイドラインに明記され、同一プロジェクトでも評価ごとに異なる場合があります。
