# ワーカープロセス・並列・並行・スレッド

## 1. 並行（Concurrent）と並列（Parallel）の違い

| | 説明 | CPUコア数 |
|---|---|---|
| **並行（Concurrent）** | 1つのCPUが時分割でタスクを切り替えながら、見かけ上同時に進める | 1コアでも可能 |
| **並列（Parallel）** | 複数のCPUコアが文字通り同時にタスクを実行する | 複数コア必要 |

```
並行（Concurrent）:
CPU: [タスクA] [タスクB] [タスクA] [タスクB] ...
     ←────────── 時間 ──────────→

並列（Parallel）:
CPU1: [タスクA] [タスクA] [タスクA] ...
CPU2: [タスクB] [タスクB] [タスクB] ...
      ←────────── 時間 ──────────→
```

---

## 2. プロセスとスレッドの違い

### プロセス
- OSから**独立したメモリ空間**を割り当てられる
- プロセス間でメモリは原則共有しない
- 並行・並列どちらも実現可能

### スレッド
- 同一プロセス内で**メモリ空間を共有**する
- 共有メモリへの同時アクセスで競合が起きるため、`mutex_lock` などで保護が必要
- 並行・並列どちらも実現可能

```
プロセスA [メモリA]    プロセスB [メモリB]
  └─ スレッド1           └─ スレッド1
  └─ スレッド2           └─ スレッド2
       ↑共有                   ↑共有
  （プロセス間は非共有）
```

---

## 3. ワーカープロセスとは

ワーカープロセスは **PHP-FPM 固有の概念ではなく**、並列・並行処理の一般的な手法である。
Apache など他のサーバーソフトウェアでも同様の考え方が使われる。

### なぜワーカープロセスが必要か

Webサーバーには複数のリクエストが同時に到着する。

```
リクエストが同時に来たとき:
  ① 順番待ちにする        → 遅い
  ② 都度プロセスをforkする → 起動コストが高い
  ③ 事前にプロセスを複数起動しておく → ワーカープロセス
```

PHP-FPM は起動時にワーカープロセスを複数立ち上げておき、
リクエストが来たら空いているワーカーに割り当てる。

```
リクエストA → ワーカー1（処理中）
リクエストB → ワーカー2（処理中）  ← 並列処理
リクエストC → ワーカー3（処理中）
リクエストD → 空き待ち（全ワーカーが埋まっている場合）
```

---

## 4. PHP-FPM がスレッドではなくプロセスを使う理由

PHP はもともとスレッドセーフな設計になっていない。
グローバル変数や拡張ライブラリがスレッド間でメモリを共有すると競合が生じるリスクがある。

プロセスを使うことでメモリ空間が完全に分離され、
mutex などを意識せずに安全に並列処理できる。

```
スレッドの場合（PHPには不向き）:
  スレッド1 ─┐
             ├─ 共有メモリ → 競合のリスク
  スレッド2 ─┘

プロセスの場合（PHP-FPMの採用方式）:
  ワーカー1 [独立メモリ]
  ワーカー2 [独立メモリ]  → 競合しない
  ワーカー3 [独立メモリ]
```

## 5. www.conf との対応

```ini
[www]                   ; プール名（ワーカープロセス群の単位）
pm = dynamic            ; ワーカー数の管理方式
pm.max_children = 5     ; ワーカープロセスの最大数
pm.start_servers = 2    ; 起動時のワーカー数
```

`pm.max_children` がワーカーの上限であり、
同時に処理できるリクエスト数の上限でもある。
