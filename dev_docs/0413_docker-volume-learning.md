# Docker Volume 学習まとめ

## 1. Named Volume と Bind Mount の違い

### データの実体の場所

| 種類 | 実体の場所 | 管理者 |
|------|-----------|--------|
| Named Volume | `/var/lib/docker/volumes/<n>/_data` | Docker |
| Bind Mount | ユーザー指定の任意のホストパス | ユーザー |

#### 一次資料からの引用

**Named Volume について（Docker Docs - Volumes）：**
> "When you create a volume, it's stored within a directory on the Docker host. When you mount the volume into a container, this directory is what's mounted into the container."

> "Volumes are completely managed by Docker"

**Bind Mount について（Docker Docs - Bind mounts）：**
> "When you use a bind mount, a file or directory on the host machine is mounted from the host into a container."

---

### 移植性（ホストのディレクトリ構成への依存）

| 種類 | 移植性 | 理由 |
|------|--------|------|
| Named Volume | 高い | Dockerが管理、ホストのディレクトリ構造に依存しない |
| Bind Mount | 低い | 指定したホストパスが存在しないと動かない |

#### 一次資料からの引用

**Named Volume の移植性（Docker Docs - Volumes）：**
> "Volumes work on both Linux and Windows containers."

**Bind Mount の移植性の問題（Docker Docs - Bind mounts）：**
> "Bind mounts rely on the host machine's filesystem having a specific directory structure available. This reliance means that containers with bind mounts may fail if run on a different host without the same directory structure."

---

### 使い分け

| 用途 | 推奨 |
|------|------|
| DBデータ（本番） | Named Volume |
| 開発時のソースコード | Bind Mount |
| 設定ファイル（開発時） | Bind Mount |

#### 一次資料からの引用

**Named Volume が推奨されるケース（Docker Docs - Volumes）：**
> "Volumes are the preferred mechanism for persisting data generated by and used by Docker containers."

**Bind Mount が適切なケース（Docker Docs - Bind mounts）：**
> "Sharing source code or build artifacts between a development environment on the Docker host and a container"

## 2. docker-compose.yml での Volume 定義

### 基本構造

```yaml
services:
  wordpress:
    volumes:
      - wordpress-volume-key:/var/www/html  # 参照名:コンテナ内パス

volumes:
  # Compose内での参照キー（Docker上の実名は name: で指定）
  wordpress-volume-key:
    name: wordpress-data       # docker volume ls で表示される名前
    driver_opts:
      device: /home/${USER}/data/wordpress
      o: bind
```

### 2つの名前の違い

| 名前 | スコープ | 用途 |
|------|---------|------|
| `wordpress-volume-key` | docker-compose.yml内のみ | YAMLのキー、Compose内での参照名 |
| `wordpress-data` | Dockerデーモン全体 | `docker volume ls` で表示される実際の名前 |

#### 一次資料からの引用（Compose file reference - Volumes）：

> "`name` sets a custom name for a volume."

### driver_opts による Bind Mount 風の動作

```yaml
driver_opts:
  device: /home/${USER}/data/wordpress  # 実体の保存場所
  o: bind                                # bind mountとして扱う
```

Named Volumeの管理機能を使いつつ、実体は指定したホストパスに保存するハイブリッドな書き方。

## 3. YAML の基本

### キーと値

```yaml
name: tanaka    # name がキー、tanaka が値
age: 25
```

### ネスト（階層構造）

インデント（スペース）で階層を表現。タブは使えない。

```yaml
services:
  wordpress:           # wordpress はキー
    image: nginx       # 値がスカラー
    ports:             # 値がリスト
      - "8080:80"
```

### キーの値がマッピングの場合

```yaml
wordpress-volume-key:      # ← 右が空欄に見える
  name: wordpress-data     # ← 実際はネストした構造が値
  driver_opts:
    device: /path/to/dir
```

`wordpress-volume-key:` の値は「`name`, `driver_opts` を含むオブジェクト全体」。

### リスト

```yaml
ports:
  - "8080:80"
  - "443:443"
```

## 4. 命名規則（Inception向け）

### 推奨パターン

```yaml
volumes:
  # Compose内での参照キー（Docker上の実名は name: で指定）
  wordpress-volume-key:
    name: wordpress-data
```

- `-key` サフィックスで「YAMLのキーであり実体の名前ではない」ことを明示
- コメントで補足（サフィックスだけでは伝わらない）
- `dummy` は避ける（「消していい？」と誤解される）

### シンプルにしたい場合

```yaml
volumes:
  wordpress-data:
    name: wordpress-data
```

参照名と実名を同じにすれば混乱が減る。
