# MariaDB コンテナ構築と接続テストの記録

## 1. 整理とクリーンアップ
不要なイメージやコンテナが溜まった際、以下のコマンドでディスク容量を解放した。
* **コマンド**: `docker system prune`
* **結果**: 約691MBのスペースを回収。
* **注意**: `bridge`, `host`, `none` ネットワークはDockerのデフォルト設定のため、削除されずに残るのが正常。

---

## 2. イメージビルドとネットワーク
イメージ削除後は、手動でネットワークの作成とイメージの再ビルドが必要。
* **ネットワーク作成**: `docker network create inception-test-net`
* **ビルド**: `docker build -t mariadb-test:task26 .`

---

## 3. コンテナ起動時の注意点
既に同名のコンテナ（停止中や作成失敗分）が存在する場合、`Conflict` エラーが発生する。
* **対処**: `docker rm mariadb` で古いコンテナを削除してから `docker run` を実行する。

---

## 4. 接続テストにおける「localhost」の罠
`docker exec` を使用してコンテナ内部から `wpuser` で接続テストを行う際、以下のエラーに直面した。
> `error: 'Access denied for user 'wpuser'@'localhost' (using password: YES)'`

### 原因の分析
MariaDBのユーザー認証は **「ユーザー名」＋「接続元（Host）」** のペアで行われる。
* **設定**: `wpuser` は `@'%'`（ネットワーク経由）として作成されている。
* **挙動**: コンテナ内部で `localhost` や `127.0.0.1` に接続しようとすると、MariaDBクライアントは高速な **UNIXドメインソケット** を使用しようとする。
* **不一致**: ソケット通信は `localhost` からの接続と見なされるが、`wpuser@'localhost'` は存在しないため拒否される。

### 認証情報の確認結果
`SELECT User, Host FROM mysql.user;` の実行結果：
| User | Host |
| :--- | :--- |
| wpuser | % |
| root | localhost |

---

## 5. ネットワーク経由のテスト（IP指定）
コンテナのIPアドレス（例: `172.18.0.2`）を直接指定した場合、エラーメッセージが変化した。
> `error: 'Access denied for user 'wpuser'@'469fd466d14b'`
※ `469fd466d14b` はコンテナID。

* **進展**: 「ソケット（localhost）」ではなく「ネットワーク（ホスト名）」での認証フェーズに移行したことを意味する。
* **結論**: `wpuser@'%'` は外部からの接続を待機する設定として**正しく機能している**。

---

## 6. 今後の展望（WordPress接続）
WordPressコンテナ（外部）から `DB_HOST=mariadb` として接続する場合、MariaDB側からは「外部ネットワークからの接続」として認識される。
これにより、今回作成した **`wpuser@'%'` の権限が正しく適用される**ため、本番環境での動作に支障はない。

---

### メモ
* パスワード指定の際は `-p` とパスワードの間にスペースを入れない（例: `-pwppassword`）。
* 内部でどうしても `ping` を通したい場合は、`root` ユーザーで行うか、`wpuser@'localhost'` を追加で作成する。
